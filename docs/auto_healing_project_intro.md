## 1. 项目背景与市场痛点

随着企业数字化基础设施的持续演进，**IT 运维正面临前所未有的挑战**：

| 挑战维度 | 传统现状 | 业务影响 |
|---------|---------|---------|
| **告警洪流** | 监控系统每天产生数千条告警，人工逐一响应 | 运维人员疲于奔命，核心问题被淹没 |
| **故障响应慢** | 平均故障响应时间（MTTR）超过 30 分钟 | 业务中断成本以分钟计 |
| **重复劳动** | 磁盘清理、服务重启等高频操作仍需人工介入 | 人力成本高昂，自动化程度低 |
| **流程不透明** | 操作全凭人脑记忆和口头协调，无法审计 | 合规风险高，知识流失严重 |
| **工具孤岛** | ITSM（工单系统）、CMDB、脚本库各自为政 | 上下文切换频繁，效率极低 |

**核心洞察**：企业不缺告警，也不缺脚本和工具；缺的是**一个能把"看到问题"智能转化为"自动解决问题"的闭环平台**。

## 2. 产品愿景与核心价值

**Auto-Healing Platform（AHS）** 是一个面向企业级运维团队的**智能自动化自愈平台**。它将监控告警（ITSM 工单）、资产配置（CMDB）、自动化脚本（Ansible Playbook）和审批流程有机整合，构建从"**发现问题 → 智能匹配 → 自动/人工执行 → 全链路追溯**"的一体化运维闭环。

```
┌────────────┐     ┌────────────────┐     ┌───────────────────┐     ┌────────────────┐
│ 外部 ITSM  │────▶│  告警接入与解析  │────▶│  智能规则匹配引擎  │────▶│  自动执行修复   │
│  监控系统   │     │  (Plugin 集成)  │     │  (Healing Rules)  │     │  + 人工审批节点 │
└────────────┘     └────────────────┘     └───────────────────┘     └────────────────┘
                                                                             │
                          ◀────────────────── 全链路审计 & 实时可观测 ────────┘
```

### 核心价值主张

- 🚀 **MTTR 降低 80%**：高频运维操作（磁盘清理、服务重启、流量切换）全面自动化
- 🧠 **零代码配置**：通过可视化 DAG 流程图构建复杂修复逻辑，无需编程能力
- 🔬 **完整审计链**：每一次操作均有不可篡改的执行日志，满足合规要求
- 🤝 **人机协作**：对高风险操作强制插入人工审批节点，平衡效率与安全
- 🔌 **开放集成**：通过标准 Plugin 框架对接 ServiceNow、Zabbix、Prometheus 等任意 ITSM/CMDB 系统

## 3. 核心功能体系

### 3.1 自愈引擎（Healing Engine）

> **平台的"大脑"——从告警到解决的全自动引擎**

自愈引擎是整个平台最核心的模块，包含三个紧密协作的子系统：

#### 3.1.1 自愈规则（Healing Rules）

自愈规则定义"**什么样的告警应该触发什么样的处置流程**"。

**核心特性：**

- **多维条件匹配**：支持对工单的任意字段进行精确匹配，包括标题、描述、优先级、类别、关联配置项等
- **嵌套逻辑组**：支持 AND/OR 嵌套组合，可构建任意复杂的条件树
  ```
  示例：(告警名称 包含 "CPU超限" AND 严重度 >= 3) OR 服务名 等于 "Payment核心链路"
  ```
- **优先级调度**：多条规则同时匹配时，优先级最高者生效，避免重复执行
- **双模触发**：
  - **🤖 自动模式（Auto）**：告警匹配后立即在后台触发流程，适用于低风险高频场景
  - **👨‍💼 手动模式（Manual）**：匹配后进入"待办中心"等待运维人员确认，适用于高风险变更
- **规则仿真测试**：在保存前可以用真实工单数据对规则进行模拟测试，每个条件结果高亮展示（绿色✓/红色✗），直观验证逻辑正确性

#### 3.1.2 自愈流程（Healing Flows）

自愈流程基于可视化 **DAG（有向无环图）** 画布设计执行路径，是平台的"**修复剧本**"。

**节点类型（Node Types）：**

| 节点 | 功能 |
|------|------|
| `host_extractor` | 从工单中提取受影响的目标主机 IP |
| `cmdb_validator` | 校验目标主机是否存在于 CMDB 且非维护状态 |
| `condition` | 条件分支，基于上下文数据决定走哪条路径 |
| `execution` | 触发 Ansible Playbook 自动化执行 |
| `approval` | 人工审批节点，可设定审批人/角色及超时时间 |
| `notification` | 发送告警通知（邮件/钉钉/Webhook） |
| `set_variable` | 设置/计算流程上下文变量 |
| `loop` | 循环节点，对批量资源逐一处理 |
| `compute` | 数据计算节点，支持表达式引擎 |

**设计亮点：**
- **动态变量映射**：使用表达式引擎（基于 `expr-lang`）从流程上下文中动态提取数据，每个执行节点的参数都可以引用上游节点的输出
- **多层变量优先级**：动态映射 > 节点静态配置 > 任务模板默认值，三层覆盖确保灵活性
- **通知节点多路分发**：单个通知节点可同时向多个渠道发送不同内容，支持精准路由（如：成功走 Webhook，失败同时发邮件+钉钉）
- **保护性删除**：被活跃流程引用的任务模板或通知渠道无法删除，防止配置损坏

#### 3.1.3 流程实例（Flow Instances）

每次流程触发后产生不可修改的**流程实例**，完整记录执行过程。

- **实时状态**：`pending` → `running` → `waiting_approval` → `completed` / `failed` / `cancelled`
- **SSE 实时推流**：执行过程通过 Server-Sent Events 实时推送到前端，节点状态变更延迟 < 200ms
- **双层日志架构**：
  - **轻量层**（`node_states` JSONB）：节点进入/完成时间戳、状态，用于仪表盘
  - **详细层**（`flow_execution_logs`）：Ansible stdout/stderr、变量解析过程、CMDB 查询结果，完整的法证级别日志
- **Dry-Run 沙箱**：支持"试运行"模式——真实执行数据提取和条件判断逻辑，但跳过 Ansible 执行和通知发送，允许运维人员在投产前充分验证

#### 3.1.4 全局调度器

- 每 N 秒（可配置，默认 10 秒）扫描未处理工单
- 按优先级顺序对每条工单逐一尝试匹配规则
- 一个工单仅匹配最高优先级的一条规则，避免重复执行冲突

### 3.2 插件集成平台（Plugin Integration）

> **连接一切外部 IT 系统的标准化适配层**

插件系统采用"**标准协议**"设计哲学：平台内部定义固定的数据规范（Incident、CMDBItem），外部系统通过灵活的**字段映射（Field Mapping）**进行对接，无需修改核心代码即可接入任意 ITSM 或 CMDB 系统。

**支持的接入类型：**
- **ITSM 插件**：从 ServiceNow、Jira Service Management、自研工单系统等拉取故障工单
- **CMDB 插件**：从 ServiceNow CMDB、Zabbix、自研资产库同步配置项数据

**核心能力：**

| 特性 | 说明 |
|------|------|
| **多种认证支持** | Basic Auth、Bearer Token、API Key，满足各类外部系统安全要求 |
| **定时轮询同步** | 可配置同步间隔（分钟级），增量拉取新增/变更数据 |
| **手动即时同步** | 一键触发立即同步，无需等待下次调度 |
| **字段映射引擎** | 可视化配置外部字段到内部标准字段的映射关系，支持 `raw_data` 嵌套字段点号路径访问 |
| **智能过滤引擎** | 基于 AND/OR 逻辑组合的过滤规则，只同步符合条件的记录（如：只拉取 severity=critical 的工单） |
| **双向写回** | 支持自愈处理完成后回写工单状态到外部 ITSM |
| **同步可观测** | 详细同步日志，精确记录拉取数、过滤数、新增数、更新数、失败数及失败原因 |
| **CMDB 维护感知** | 同步完成后自动联动检查维护期到期的资产，还原其状态 |

**可视化配置界面：**
- 创建阶段：**4 步向导**（基础信息 → 连接配置 → 同步设置 → 字段映射）
- 编辑阶段：**标签页式表单**，直接跳转到需要修改的配置区块，效率优先

### 3.3 CMDB 资产管理

> **平台内置的资产配置库——链接告警与执行的关键枢纽**

CMDB 模块是自愈引擎中 `cmdb_validator` 节点的数据来源，确保自动修复始终针对真实存在且状态正常的资产。

**核心设计：**

- **3 态生命周期**（简洁无歧义）：
  - `active`：正常运行，可被纳入自愈流程
  - `offline`：已下线/退役，不可触碰
  - `maintenance`：维护保护中，自动跳过修复操作，避免"维护中修复"冲突
  
- **维护窗口管理**：
  - 支持批量设置维护状态（100+ 台主机一键进入维护模式）
  - 可选设置预计恢复时间，系统**自动到时解除维护状态**
  - 所有维护操作记录完整的操作日志（操作人、原因、时间）

- **资产编排工作台**：
  - 双面板设计（左侧拓扑树 + 右侧资产列表）
  - 支持按 IP、主机名、操作系统、所属部门、环境多维过滤
  - 支持 2000+ 节点规模的流畅选择操作

- **自愈排除机制**：处于 `offline` 或 `maintenance` 状态的资产，在流程执行中自动被标记跳过，并记录具体的跳过原因

### 3.4 Git 仓库 & Playbook 管理

> **将 Ansible 自动化代码库纳入规范化治理**

这一模块实现了从"版本控制代码"到"可执行修复脚本"的规范化管理链路。

**层次结构：**
```
Git Repository（代码仓库）
    └── Playbook Template（自动化蓝图）
            └── Execution Task Template（任务模板）
                        └── Execution Run（执行记录）
```

**Git 仓库管理：**
- 支持注册外部 Git 仓库，自动克隆到本地
- 分支管理：注册后分支锁定，防止意外变更执行环境
- 定时自动同步（Git Pull），保持与远端代码最新一致
- 完整的同步日志记录

**Playbook 模板（自动化蓝图）：**

| 功能 | 说明 |
|------|------|
| **智能变量扫描** | 递归分析 Playbook 文件（含 roles、tasks 引用的子文件），自动发现 Jinja2 变量列表 |
| **两种扫描模式** | `auto`（启发式自动发现）+ `enhanced`（基于 `.auto-healing.yml` 配置文件的精确扫描） |
| **3 态生命周期** | `pending`（待扫描）→ `scanned`（已分析）→ `ready`（可用于生产） |
| **漂移检测** | 代码变更后重新扫描时，自动对比变量差异，标记所有关联任务模板为"需审核"状态 |
| **强制审核封锁** | 被标记为漂移的任务模板，**严禁触发任何执行**，直至运维人员确认变更 |
| **变量精细化** | 可对每个变量添加说明、设置默认值约束、标记是否为敏感字段 |

### 3.5 执行中心（Execution Center）

> **"任务发射台"——支持手动、计划、自愈三种触发模式的统一执行引擎**

**执行任务模板（Task Template）**是所有执行的最小单元，封装了 Playbook 路径、目标主机、变量预设和通知配置。

**三种触发模式：**

```
手动触发（LAUNCH PAD）
    ──── 操作员主导，即时选择目标+变量后执行
计划触发（ORBIT CONTROL）
    ──── 配置 Cron 表达式定时自动触发，支持独立的参数覆盖
自愈触发（Healing Flow）
    ──── 由自愈流程的 "execution" 节点自动调用，目标主机由 CMDB/工单动态注入
```

**执行时参数覆盖：**
- **目标主机**：全量替换模板预设（确保自愈场景每次针对实际故障主机）
- **变量覆盖**：运行时变量与模板默认值合并，运行时优先
- **通知控制**：支持静默执行（`skip_notification`）和临时通知接收人追加

**实时可观测性（SSE 流）：**
- 执行启动后立即跳转至"实时日志驾驶舱"
- 通过 Server-Sent Events 持续推送 Ansible 执行输出（stdout/stderr）
- 执行结果采用**三态判定**标准（成功/部分成功/失败），防止"Ansible 返回 0 但所有主机均不可达"的虚假成功

**执行记录（Execution Run）：**
- 统一记录所有触发来源（`manual` / `scheduler:cron` / `scheduler:once` / `healing:xxx`）
- 支持按执行人、触发来源、时间窗口、状态多维检索
- 计划任务历史与手动执行历史统一视图，拥有完整的法证一致性

### 3.6 通知管理（Notification Management）

> **企业级多渠道告警分发引擎**

通知模块将"触发器"与"送达机制"完全解耦，实现高度灵活的告警路由。

**三大组件：**

**① 通知渠道（Channels）**：
- `Email`：标准邮件通知
- `DingTalk`（钉钉）：企业即时通讯
- `Webhook`：通用 HTTP 推送，对接 Grafana、PagerDuty 等三方系统
- 支持 Basic Auth 增强安全性
- 可配置启用/禁用及设为系统默认渠道

**② 通知模板（Templates）**：
- 模板支持引用平台标准变量（工单信息、执行结果、系统上下文）
- 变量渲染实时预览——在保存前可即时查看最终消息样式
- 渠道兼容性控制：模板声明支持的渠道类型，防止邮件模板误发至 Webhook

**③ 通知日志（Notification Logs）**：
- 完整的发送追踪（`pending` / `sent` / `failed` / `delivered` / `bounced`）
- 关联原始触发源（Execution Run / Flow Instance / Incident）
- 失败自动重试（指数退避策略）+ 分级限流
- 法证级日志：记录服务商原始响应、重试时间点等调试信息

**智能配置矩阵：**
- 执行任务支持**按状态独立配置通知**：执行开始/成功/失败分别路由不同渠道和模板
- 自愈流程通知节点支持**多对多渠道-模板组合**，实现精准的多路分发

---

### 3.7 待办决策中心（Pending Center）

> **人机协作的核心工作台——运维人员的"决策驾驶舱"**

待办中心是平台中所有**需要人工介入**决策的统一入口，分为两大工作区：

**① 审批工作台（Approvals）**：
- 来源：`approval` 节点到达时自动创建审批任务
- 展示审批上下文（关联工单、流程名称、发起人）
- 支持**批准/拒绝**，并可填写审批意见
- 超时自动拒绝（可在流程中配置超时时间）
- 完整审批日志持久化（谁批的、什么时候批的、批了什么评论）

**② 手动触发工作台（Manual Triggers）**：
- 来源：`触发模式=手动` 的自愈规则命中工单后进入此队列
- 运维人员确认工单信息后手动点击触发，流程随即启动
- 防止误触发保护：手动触发场景对工单详情完整展示

两个工作区共享统一的消息推送机制——审批任务创建时自动通过通知模块向指定审批人发送消息。

### 3.8 密钥管理（Secrets Management）

> **集中式凭证治理——安全是第一优先级**

平台内所有涉及外部系统连接的凭证（SSH 私钥、API Token、服务账号密码）均通过 **Secrets Management** 模块统一管理。

**核心特性：**
- 支持 `SSH Key`、`Webhook Token`、`API Key` 多种凭证类型
- 密钥与脚本完全分离，Playbook 通过引用凭证 ID 使用，绝不以明文嵌入代码
- 安全约束：必须将凭证置为禁用状态才能删除，防止意外断开关联配置
- 多来源分层选择策略：平台密钥库 > 项目级密钥 > 临时注入，满足不同隔离需求
- 秘钥选择器（v3）支持在任何配置界面快速检索和切换凭证

### 3.9 用户与权限管理（RBAC）

> **精细化角色权限控制，满足企业安全合规**

**RBAC 治理模型：**

```
用户（User）─── 属于 ──▶ 角色（Role）─── 拥有 ──▶ 权限（Permission）
```

**权限粒度细化至操作级别：**

| 资源 | 可配置权限 |
|------|---------|
| 自愈规则 | view / create / update / delete |
| 自愈流程 | view / create / update / delete |
| 执行任务 | view / execute / admin |
| 通知渠道 | view / manage |

**设计亮点：**
- **双列转移组件**：直观的权限分配界面（已有权限 ↔ 可授权限）
- **系统角色保护**：内置角色（super_admin、admin、operator）防止误删
- **应用账号（App Account）**：为非人类身份（CI/CD 流水线、运维脚本）提供 Access Key，独立权限管理
- **操作审计（Audit Log）**：所有管理操作、审批操作均记录到 `audit_logs`，满足合规审计

> **对接企业 AD/SSO，实现无缝员工身份接入**

- 支持 SAML 2.0 协议，可对接 **ADFS、Azure AD、阿里云 SSO** 等主流 IdP
- **JIT 自动开户**：员工首次 SSO 登录时自动在平台创建账号，零人工干预
- **5 种授权策略**：开放、白名单、域名、角色映射、混合策略——满足从初创到大型企业的不同安全需求
- **SAML 属性自动角色分配**：通过映射 SAML 的 `memberOf` 等属性自动赋予平台角色，消除人工配置工作量

## 5. 设计哲学与工程标准

### 5.1 "后端权威"原则（Backend Authority）

前端永远是后端状态机的**高保真显示器**，不在客户端独立实现业务状态转换。这一设计确保：
- 数据一致性（Consistency）：刷新页面不会出现状态闪烁
- 单一事实来源（Single Source of Truth）：后端数据库永远是权威

### 5.2 零盲区法证日志（No-Blind-Spot Forensics）

每个节点执行必须产生至少一条 `Info` 级别日志，每次失败必须记录具体原因。  
这意味着任何线上问题都可以通过日志完整复现，没有"黑盒"执行。

### 5.3 工业级 UI 设计系统（Industrial Design System）

平台前端遵循严格的**工业仪表盘设计规范（IDS）**：
- 无圆角几何感（`borderRadius: 2px`），营造专业"运维驾驶舱"质感
- 64px 元素基准高度，拇指友好的触控体验
- 统一 `[图标] 中文 / ENGLISH` 页面命名规范
- 200ms 防抖搜索，所有文本输入零延迟响应

### 5.4 人机协作安全设计

- **高风险操作必须人工确认**：大规模批量操作、高优先级变更强制插入审批节点
- **规则须绑定流程才能激活**：防止"孤儿规则"（有触发条件但无执行动作）造成预期之外的行为
- **资产维护态排除修复**：避免在人工维护期间触发自动化操作，防止冲突

## 7. 典型使用场景

### 场景一：磁盘空间告警自动清理

```
1. Zabbix 触发"磁盘使用率 > 85%"告警
2. Plugin 同步告警为工单，提取受影响主机 IP
3. 自愈规则命中（类型="存储"，级别>=中），触发"磁盘清理流程"
4. host_extractor 提取 IP → cmdb_validator 确认主机状态正常
5. execution 节点执行 cleanup.yml，清理日志和临时文件
6. 执行完成 → notification 节点发送"磁盘清理完成"通知至钉钉
7. 整个过程无人工介入，MTTR < 2 分钟
```

### 场景二：核心服务异常自动重启（含审批保护）

```
1. 监控检测到 Payment 服务崩溃，创建高优先级工单
2. 自愈规则命中，trigger_mode=manual（高风险，等待审批）
3. 审批通知推送至值班负责人钉钉账号
4. 负责人在待办中心审核工单详情，点击"批准"
5. 流程继续：cmdb_validator 确认服务主机 ≠ 维护状态
6. execution 节点执行 restart_service.yml
7. condition 节点判断重启结果：成功→通知，失败→上报高级告警
```

### 场景三：多数据中心批量合规扫描

```
1. 定时任务每周五晚间自动触发合规检查 Playbook
2. 目标主机从 CMDB 动态查询"生产环境 + active 状态"全部服务器
3. 并行执行 compliance_check.yml，汇总结果
4. 按执行结果分两路：  
   - 合规主机 → 记录成功日志  
   - 不合规主机 → 发送邮件 + 创建修复任务
5. 所有操作均记录在执行历史中，可按周导出合规报告
```
