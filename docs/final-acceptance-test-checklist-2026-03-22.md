# 最终验收测试清单

更新时间：2026-03-22

本文档用于本轮整改后的最终业务验收测试。

目标不是继续“扫代码”，而是验证真实业务链路是否符合预期，并在发现问题后做定点修复与补测。

## 测试原则

- 先执行数据库迁移：`051` 到 `054`
- 优先验证真实业务链路，而不是只看接口返回码
- 每个场景同时关注：
  - 接口返回
  - 页面表现
  - 数据库最终状态
  - 日志/SSE 是否收口
- 发现问题后，记录：
  - 触发步骤
  - 实际结果
  - 期望结果
  - 相关租户 / 用户 / 任务 ID

## 一、登录与 Token

### 1. 登录

验证点：

- 平台用户可正常登录
- 租户用户可正常登录
- 多租户用户登录后返回稳定的 `current_tenant_id`
- 返回的租户列表顺序稳定

预期：

- 登录成功后返回 `access_token`、`refresh_token`
- 多租户用户多次登录，默认租户不随机漂移

### 2. Refresh Token

验证点：

- 使用有效 `refresh_token` 可以刷新成功
- 同一个 `refresh_token` 第二次重复使用应失败
- 已禁用用户无法刷新 token

预期：

- refresh 成功后旧 refresh token 失效
- 重放旧 refresh token 返回未授权

### 3. Logout

验证点：

- 仅传 access token 登出
- 同时传 `refresh_token` 登出

预期：

- access token 失效
- 若携带 `refresh_token`，refresh token 也失效

## 二、多租户上下文与 `/common/*`

### 1. 默认租户行为

验证点：

- 多租户用户不传 `X-Tenant-ID` 访问 `/api/v1/common/*`
- 多租户用户显式传 `X-Tenant-ID` 访问 `/api/v1/common/*`

预期：

- 不传时使用稳定默认租户
- 显式传入时切换到对应租户
- 搜索、偏好、收藏、最近访问、工作台数据都随租户变化

### 2. 平台用户 Common 路由

验证点：

- 平台用户在非 Impersonation 状态访问 `/api/v1/common/search`
- 平台用户在 Impersonation 状态访问 `/api/v1/common/search`

预期：

- 非 Impersonation 状态下，不应误落默认租户数据
- Impersonation 状态下，应使用目标租户上下文

## 三、租户用户与角色边界

### 1. 租户用户管理

验证点：

- 租户管理员查询本租户用户详情
- 更新本租户用户基础信息
- 更新本租户用户角色
- 删除本租户成员
- 重置本租户成员密码

预期：

- 仅能操作当前租户成员
- 非法角色时，整个更新请求失败且不出现部分字段已更新
- 删除成员后仅移出当前租户，不应删除全局账号

### 2. 跨租户隔离

验证点：

- 租户 A 管理员尝试访问租户 B 用户/角色
- 租户 A 管理员尝试修改租户 B 自定义角色

预期：

- 返回无权限或资源不存在
- 不得跨租户生效

## 四、Dashboard 配置与工作区

### 1. Dashboard 配置隔离

验证点：

- 同一用户在租户 A 保存 Dashboard 配置
- 同一用户在租户 B 保存不同 Dashboard 配置
- 来回切租户读取配置

预期：

- 两个租户配置互不覆盖
- 返回的配置严格对应当前租户

### 2. 工作区权限

验证点：

- 用户在租户 A、B 拥有不同角色
- 查看当前租户可见工作区
- 创建新系统工作区

预期：

- 当前租户只使用当前租户角色做授权判断
- 不应因为别的租户有高权限角色而看到当前租户不该看的工作区
- 新建工作区后自动分配只绑定当前租户角色

### 3. 默认工作区

验证点：

- 默认工作区对当前租户可见
- 非默认工作区只对授权角色可见

预期：

- 工作区可见性符合角色配置

## 五、自愈流程：审批、取消、重试、超时

### 1. 审批节点

验证点：

- 触发进入审批节点
- 查看审批任务列表
- 审批通过
- 审批拒绝
- 重复点击审批通过/拒绝

预期：

- 审批任务创建成功，流程实例进入 `waiting_approval`
- 审批只能成功一次
- 重复审批不能再次恢复流程

### 2. 取消待审批流程

验证点：

- 流程停在审批节点时手动取消
- 取消后再查看审批任务
- 取消后再尝试审批

预期：

- 流程实例变成 `cancelled`
- 审批任务同步关闭，不再保持 `pending`
- 取消后审批不能再成功推进流程

### 3. 审批超时

验证点：

- 创建短超时审批任务
- 等待超时任务扫描

预期：

- 仅仍处于 `waiting_approval` 的流程会变 `failed`
- 已取消流程不会被超时扫描改回 `failed`

### 4. 重试

验证点：

- 对失败流程发起重试
- 并发或重复点击重试

预期：

- 只应有一次有效重试
- 不应出现同一实例被重复执行两遍

## 六、执行任务与日志 SSE

### 1. 执行 run 状态

验证点：

- 正常成功
- 完全失败
- 手动取消
- 部分成功（`partial`）

预期：

- run 状态与真实执行结果一致
- `cancelled` 不会被后续结果覆盖
- `partial` 被视为终态

### 2. 执行日志 SSE

验证点：

- 连接 `/execution-runs/:id/stream`
- 分别覆盖 success / failed / cancelled / partial

预期：

- 终态时发送 `done`
- `partial` 也必须收口
- 前端不会一直停留在“执行中”

## 七、流程事件 SSE

### 1. 运行中事件流

验证点：

- 正常运行中的流程连接 `/events`
- 节点开始、完成、流程完成

预期：

- 事件顺序合理
- 流程终态时连接关闭

### 2. 取消场景

验证点：

- 运行中取消
- 待审批取消
- 对已取消实例重新连接 `/events`

预期：

- 取消时会收到流程终态事件
- 已终态实例连接后应立即收口
- 不应长时间挂起 SSE

## 八、调度器与执行结果

### 1. Cron 调度

验证点：

- 成功执行
- 失败执行
- 部分成功执行
- 连续失败直到自动暂停

预期：

- 失败计数以真实执行结果为准
- 成功时重置失败计数
- 达到阈值后自动暂停

### 2. Once 调度

验证点：

- once 任务执行成功
- once 任务执行失败

预期：

- 不应在 run 尚未结束时就标记 completed
- 最终状态与真实执行结果一致

## 九、插件 / Git / Playbook 业务链路

### 1. 插件同步

验证点：

- 全部成功
- 部分记录入库失败

预期：

- 若存在记录级失败，同步日志不能记成全成功
- 不应推进 `LastSyncAt` 导致失败数据被跳过

### 2. Git 仓库同步

验证点：

- 正常克隆/拉取成功
- 模拟最终仓库状态更新失败

预期：

- 成功后仓库状态、`LastSyncAt`、`LastCommitID` 一致
- 若最终持久化失败，不应只留下“成功日志”而仓库状态卡死

## 十、站内信 / 偏好 / 收藏 / 最近访问

### 1. 站内信已读状态

验证点：

- 同一用户在租户 A 标记已读
- 在租户 B 查看同一类消息

预期：

- 已读状态按租户隔离

### 2. 用户偏好

验证点：

- 同一用户在不同租户保存不同偏好

预期：

- 偏好互不覆盖

### 3. 收藏与最近访问

验证点：

- 多租户用户在不同租户下操作收藏和最近访问

预期：

- 数据按租户隔离
- 默认租户和显式租户切换后表现一致

## 十一、日志与安全

### 1. Query Token

验证点：

- 普通 API 使用 `?token=...`
- SSE API 使用 `?token=...`

预期：

- 普通 API 不接受 query token
- SSE 端点仍可用

### 2. 日志脱敏

验证点：

- 访问带 `token` / `access_token` / `refresh_token` 的请求

预期：

- 日志中对应参数被脱敏

## 验收通过标准

满足以下条件即可认为本轮代码整改阶段完成：

- 上述关键场景无阻塞性业务错误
- 不再出现跨租户串数据、越权、状态回写覆盖、SSE 悬挂
- `go test ./...` 通过
- `go vet ./...` 通过
- migration 执行成功且历史数据可用

## 建议执行顺序

1. 先跑登录/租户切换/公共接口
2. 再跑租户用户/角色边界
3. 再跑 Dashboard/工作区
4. 再跑自愈流程审批/取消/超时/重试
5. 再跑执行 run / SSE / 调度器
6. 最后跑插件、Git、站内信、偏好和日志脱敏
